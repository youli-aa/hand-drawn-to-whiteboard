# 手绘转飞书画板

将手绘结构图/脑图照片或文字描述 → 专业结构图 → 飞书云文档画板（手机端可编辑）。

## 输入类型

| 类型 | 触发方式 | 流程差异 |
|------|---------|---------|
| **手绘图片** | 用户上传图片（照片/截图）+ "转成画板/生成结构图/做成脑图"等 | 第 1 步先读图识别节点/连线/分组/文字/层级 |
| **文字描述** | 用户描述结构/流程/概念（如"画一个用户增长飞轮""软件设计全流程结构图"） | 跳过第 1 步识别，直接从第 2 步生成 HTML |

> 两种输入最终都走同一条链路：HTML 生成 → SVG 转换 → 飞书画板。无论输入是什么，交付物都是飞书云文档画板 URL。

## 依赖

**使用本技能前，必须先安装以下技能，否则第 2 步会失败：**

| 依赖技能 | 用途 | 安装方式 |
|---------|------|---------|
| `diagram-maker` | 生成专业结构图 HTML | 在 Aily 技能市场中搜索安装 |

> ⚠️ 如果 diagram-maker 未安装，技能会回退到手动构造 SVG，但效果不如 diagram-maker 生成的图专业。

## 流程

### 1. 读取输入

**判断输入类型，按分支处理：**

**手绘图片输入：**

用户上传的图片在 `attachments/` 目录下。用 `file` 工具读取该图片，提取内容：

```
file read <attachment_path>
```

**识别必须完整，不能遗漏：**

- **实体（节点）**：每一个方块、圆圈、椭圆，无论大小、是否带文字。一个节点都不能漏。
- **连接线 / 箭头**：每条线、每个箭头、每条带箭头的连线都必须识别出来，包括单向/双向、有箭头/无箭头（实线 vs 虚线、实箭头 vs 空心箭头）。
- **方向**：箭头从哪里出发，指向哪里。
- **分组 / 区域**：虚线框、阴影区、背景色块——这些不是装饰，是"子流程/待办/未列全"等语义标注。
- **文字标注**：节点上的文字、连接线上的文字、区域标题。
- **层级关系**：父子、并列、主干-分支、汇总。

**最终清点：**
- 列出全部节点
- 列出全部连接（起点 → 终点，含箭头方向、线型）
- 列出全部分组
- 比对原图，确保每一处都对应到 SVG 里的一个元素

**文字描述输入：**

用户通过自然语言描述了结构/流程/概念。跳过图片识别，直接将用户的文字描述作为第 2 步的输入。注意：
- 如果用户描述模糊（如"画一个架构图"但没说是哪层架构），先追问一两句澄清，再进入第 2 步
- 如果用户描述清晰，直接进入第 2 步

### 2. 生成结构图 HTML

调用 `get_skills("diagram-maker")` 加载 diagram-maker 技能，按其指引生成结构图 HTML。

**无论输入是手绘图还是文字描述，第 2 步统一调 diagram-maker 生成 HTML。**

**要点：**
- 手绘图片输入：保持原图的结构逻辑，不做额外发挥
- 文字描述输入：根据用户描述的结构/流程/概念生成图，如有歧义先澄清
- 原图中标注"未列全""……"的地方，在图中标注"（未列全）"
- 使用语义色区分不同类型节点（起始/主干/收尾/分支）
- 输出 HTML 文件到 artifacts 目录

### 3. HTML → SVG

运行转换脚本：

```bash
python3 <skill_dir>/scripts/html_to_svg.py <input.html> <output.svg>
```

**如果脚本失败（HTML 不含内嵌 SVG）：** 手动构造 SVG。读取 HTML 结构，用标准 SVG 元素（`<rect>`、`<text>`、`<path>`、`<g>`）重建，确保：
- 根元素 `<svg>` 包含 `xmlns="http://www.w3.org/2000/svg"` 和 `viewBox`
- 所有 `<text>` 元素包含 `font-family="Noto Sans SC, sans-serif"`
- 无 `<style>` 标签、CSS class、`data-*` 属性
- 参考 `references/whiteboard-guide.md` 的 SVG 格式要求

### 4. 写入飞书画板

**主方案（一步到位）：** 创建文档时直接嵌入 SVG 画板。

```bash
lark-cli docs +create --title "<图表标题>" \
  --content '<whiteboard type="svg"><svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 800 600">...</svg></whiteboard>' \
  --as user
```

> SVG 内容需要先从第 3 步生成的 `.svg` 文件中读取，整体嵌入 `<whiteboard type="svg">` 标签。

**备选方案：** 如果 `lark-cli docs +create` 不支持直接嵌入 SVG 画板，回退到 `references/whiteboard-guide.md` 中的传统 4 步流程。

**写入前自检清单（避免遗漏）：**
- [ ] 每个节点都有对应元素
- [ ] 每条连接线都有对应 `<path>`
- [ ] 箭头方向正确
- [ ] 每个分组区域都有对应容器（虚线框/背景色块）
- [ ] 所有文字标注都已包含
- [ ] 手绘输入：比对原图，确保每一处都对应到 SVG 里的一个元素

### 5. 交付

**只返回最终的飞书画板链接**（飞书文档 URL），不展示 HTML/PNG/SVG 等中间产物。

回复示例：
```
> https://beike.feishu.cn/docx/...

流程图已放在云文档的画板里了，双击画板区域即可进入编辑模式，可以拖动节点、改文字、调颜色。
```

**回复中必须包含的提示：**
- 告诉用户"双击画板区域即可编辑"——很多用户不知道飞书画板怎么进入编辑模式
- 简要说明可以做什么（拖动节点、改文字、调颜色），让用户知道这不是静态图片

**不要在回复中：**
- 列中间产物文件路径
- 贴 HTML 代码片段
- 给 PNG 链接
- 解释每一步在干什么（除非出问题）

## 注意事项

- 如果 diagram-maker 不可用，回退到手动构造 SVG
- 飞书画板不支持复杂 CSS，SVG 必须使用 presentation attributes（`fill="..."` 而非 `style="fill:..."`）
- 写入画板前，确保 `cd` 到 SVG 所在目录，使用相对路径 `@./file.svg`
- 每个步骤完成后检查结果，失败时不要跳过，按参考文档排查
